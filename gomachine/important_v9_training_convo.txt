⏺ IIR = Internal Iterative Reduction. It's a move-ordering trick. When the
  search reaches a node that should be searched to depth D but has no TT move
  (the transposition table has nothing stored for this position), the move
  ordering there is bad — you don't know which move is best, so you'd waste a
  big full-depth search trying them in a poor order.

  IIR's response: if there's no TT move, search this node one ply shallower 
  (D−1). Two payoffs:
  1. Cheaper — you don't sink a full-depth search into a node you can't order
  well.
  2. It seeds the TT — the shallower search stores a best move, so when this
  node (or a sibling/re-search) is visited again, you now have a good move to
  try first → more beta cutoffs → smaller tree.

  It's the modern, lighter successor to Internal Iterative Deepening (which did
  a whole reduced search first to find a move); IIR just says "no hint → reduce
  a ply, and the result becomes the hint."

  Why it's helping now when it didn't before: the old version fired on every 
  node type and over-reduced (−33.7 Elo — it shaved depth at cut-nodes/all-nodes
  where that's just lost depth). The rework fires only on PV nodes (the
  Stockfish-style placement), where seeding a good TT move actually pays and the
  reduction is safe. Same idea, correct location → net positive instead of net
  negative. It reinvests search effort from poorly-ordered subtrees into
  well-ordered ones.

  v6 vs v9 — brief

  They're the same architecture; v9 just adds one thing. Both are:

  inputs → Feature Transformer → accumulator → tail → bucketed output

  ┌──────────┬──────────────┬─────────────────────────────┐
  │          │      v6      │             v9              │
  ├──────────┼──────────────┼─────────────────────────────┤
  │ inputs   │ 768          │ 768 + 9,216 threats = 9,984 │
  ├──────────┼──────────────┼─────────────────────────────┤
  │ FT width │ 512          │ 512                         │
  ├──────────┼──────────────┼─────────────────────────────┤
  │ tail     │ single-layer │ single-layer                │
  ├──────────┼──────────────┼─────────────────────────────┤
  │ buckets  │ 8            │ 8                           │
  └──────────┴──────────────┴─────────────────────────────┘

  The terms:

  - Inputs / features — what the net "sees." The 768 are piece-square: "is there
  a [white knight] on [e5]?" — 12 piece types × 64 squares. Just where the 
  pieces are.
  - Threats (v9's only addition) — extra inputs that say which piece attacks 
  which occupied square: "this knight attacks that bishop on c4." v6 has to
  re-derive attacks internally from raw placement; v9 is told them directly → a
  stronger eval per neuron (that's the +115 fixed-depth edge). Cost: 9,216 of
  them, so the accumulator update is heavier (the NPS deficit).
  - Feature Transformer (FT) — the big first layer. Maps the sparse active
  inputs to a dense 512-number hidden vector. It's incremental: a move only
  updates the changed inputs (fast), instead of recomputing from scratch.
  - Accumulator — the FT's running output, kept up to date move-by-move. Two of
  them — one from White's view, one from Black's ("×2 perspectives") — combined
  at eval time.
  - Tail — everything after the accumulator that turns those 512 numbers into a
  single centipawn score. "Single-layer" = just activate (SCReLU) → one dot 
  product → number. (A multilayer tail would add hidden layers L1/L2 — more eval
  power, slower; that's a future rung. The tail is what I fused to integer math
  for the NPS win.)
  - Buckets — 8 separate output "heads," picked by material count. The right
  eval emphasis differs opening vs endgame, so the net uses a phase-appropriate
  head. Both have 8.

  One-line version: v9 = v6 + threat inputs (and trained longer, 320-sb) — same
  width, same single-layer tail, same 8 buckets. Richer eyes, same body.
